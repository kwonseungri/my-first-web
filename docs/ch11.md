# Chapter 11. Row Level Security (RLS) 적용 가이드

본 문서는 내 블로그(**my-first-web**)의 데이터베이스 보안을 강화하기 위해 **Row Level Security (RLS)** 정책을 설계하고 적용하는 가이드입니다. 

실제 RLS 정책은 Supabase SQL Editor에서 수동으로 실행하는 대신, **Supabase CLI 마이그레이션 파일**로 남겨 안전하게 버전 관리를 수행합니다.

---

## 1. RLS 핵심 보안 원칙

1. **보안의 주체는 데이터베이스**: 
   - 프론트엔드 UI 상에서 본인의 글에만 수정/삭제 버튼이 보이는 것은 단순한 **UX 개선**입니다.
   - 실제 비인가 사용자의 API 호출(게시글 수정/삭제 요청)을 차단하는 **실제 보안**은 오직 **Supabase RLS 정책**이 보장합니다.
2. **서비스 역할 키(`service_role`) 사용 금지**:
   - `service_role` 키는 RLS를 우회할 수 있는 강력한 관리자 키입니다.
   - 클라이언트(브라우저) 코드에는 절대 노출하지 않으며, 오직 `anon` 키만을 활용하여 요청을 보내고 RLS 정책을 통과해야 합니다.
3. **마이그레이션 도구 활용**:
   - RLS 활성화 및 정책 추가는 모두 `supabase migration new` 명령어로 생성된 SQL 파일에 작성하여 `db push`로 반영합니다.

---

## 2. RLS 보안 적용 대상 및 세부 정책

우리 프로젝트의 핵심 테이블인 `profiles`와 `posts`에 대해 RLS를 활성화하고, 다음과 같이 정책(Policy)을 적용합니다.

### ① `profiles` (사용자 프로필) 테이블

* **RLS 상태**: `ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;`
* **세부 정책**:

| 작업 (Action) | 정책 조건 (USING / WITH CHECK) | 정책 설명 |
| :--- | :--- | :--- |
| **SELECT** (조회) | `true` (무조건 허용) | 블로그 글 작성자의 정보(닉네임 등)는 비로그인 방문자를 포함한 **모든 사용자가 조회**할 수 있어야 합니다. |
| **UPDATE** (수정) | `auth.uid() = id` | 로그인한 사용자는 **오직 자신의 프로필 데이터만 수정**할 수 있어야 합니다. (자신의 `uuid`가 해당 profile의 `id`와 일치해야 함) |
| **INSERT** (추가) | (인증 흐름 내에서 트리거로 처리되거나 불필요) | 회원가입 시 Supabase Auth의 트리거 함수 등을 통해 관리되므로 클라이언트 직접 생성은 기본적으로 제한하거나 동일 규칙을 따릅니다. |
| **DELETE** (삭제) | `false` (무조건 차단) | 회원이 자신의 프로필 데이터를 직접 삭제하는 것은 차단합니다. |

---

### ② `posts` (게시글) 테이블

* **RLS 상태**: `ALTER TABLE posts ENABLE ROW LEVEL SECURITY;`
* **세부 정책**:

| 작업 (Action) | 정책 조건 (USING / WITH CHECK) | 정책 설명 |
| :--- | :--- | :--- |
| **SELECT** (조회) | `true` (무조건 허용) | 블로그 게시글은 비로그인 방문자를 포함한 **모든 사용자가 조회**할 수 있어야 합니다. |
| **INSERT** (작성) | `auth.uid() = user_id` | 로그인한 사용자만 글을 작성할 수 있으며, 새로 추가하는 posts 행의 `user_id`는 **글을 쓰고 있는 로그인 사용자의 ID(`auth.uid()`)와 반드시 일치**해야 합니다. |
| **UPDATE** (수정) | `auth.uid() = user_id` | 로그인한 사용자 중 **자신이 작성한 글(`user_id`가 본인 `uuid`와 일치)만 수정**할 수 있어야 합니다. |
| **DELETE** (삭제) | `auth.uid() = user_id` | 로그인한 사용자 중 **자신이 작성한 글(`user_id`가 본인 `uuid`와 일치)만 삭제**할 수 있어야 합니다. |

## 3. RLS 적용 확인 시나리오 (검증 계획)

마이그레이션이 반영된 이후 다음 시나리오를 통해 RLS 정상 작동을 검증합니다.

1. **비로그인 사용자 테스트**:
   - `/posts` 목록 및 상세 페이지가 정상적으로 조회되는지 확인 (SELECT 허용 검증)
   - 직접 API나 다른 수단으로 글 쓰기/수정/삭제 시도 시 `401 Unauthorized` 또는 RLS 차단 에러가 발생하는지 확인
2. **로그인 사용자 A 테스트**:
   - 자신이 작성한 글을 정상적으로 수정 및 삭제할 수 있는지 확인
   - 다른 사용자 B가 작성한 글을 수정/삭제하려고 시도(네트워크 요청 강제 발송 등)할 때 데이터베이스 단에서 거부(Error: Row-level security policy violated)되는지 확인
